# Page Architecture

## Principle

Separate first-level navigation from second-level workflow screens. A good product plan includes both.

## First-Level Pages

First-level pages are stable destinations in main navigation:

- Dashboard, Today, Overview, Home.
- List, Library, Inbox, Orders, Records.
- Progress, Analytics, Reports, Calendar.
- Profile, Settings, Account.

First-level pages alone are not a product flow.

## Second-Level Pages

Second-level pages complete the work:

- Create or start page.
- Active task page.
- Detail page.
- Edit page.
- Add item sheet or form.
- Confirmation page.
- Result or summary page.
- Settings detail page.
- Integration permission page.

These pages make the product operational.

## Inventory Format

Use this format:

```text
First-level pages
1. Page name: purpose, primary content, entry points.

Second-level pages
1. Page name: parent page, trigger, purpose, primary content.
```

## Depth Check

Before approving the architecture, check:

- Does the user have a place to start?
- Can the user create or perform the core action?
- Can the user inspect details?
- Can the user edit or correct inputs?
- Is there a result or feedback screen?
- Can the user review history or progress?
- Can the user change preferences/goals that shape the next loop?

If not, add the missing second-level page.
